### What I fixed

1. **`fetch()` → `api.fetch()`** — The original used the browser's native `fetch()` directly, which gets blocked by CORS when calling external APIs. `api.fetch()` routes through the dashboard's built-in proxy, bypassing that restriction.

2. **`document.getElementById()` → `container.querySelector()`** — Scoped all DOM lookups to the widget's container instead of the whole document, which is safer if multiple widgets are on the same page.

The API is working — you're just hitting Finnhub's **rate limit** (60 requests/minute on the free tier). With 5 stocks refreshing every 30 seconds, requests pile up fast. I'll fix this by:

1. **Adding delay between requests** (already there but too short)
2. **Slowing the refresh rate** and staggering calls more
3. **Caching last good data** so a 429 doesn't wipe the display
4. **Showing a rate-limit warning** instead of silently failing

### What changed

| Fix | Detail |
|-----|--------|
| **Min refresh = 60s** | `Math.max(60, ...)` prevents accidentally setting an interval too short for Finnhub's free tier |
| **1s gap between requests** | Increased from 300ms → 1000ms, so 5 stocks takes 5s total — well within 60 req/min |
| **429 detection** | Checks `r.status === 429` specifically and throws `'RATE_LIMIT'` |
| **In-memory cache** | Last good quote is cached per symbol — a 429 shows stale data instead of blanking |
| **Early loop exit on 429** | Stops firing more requests immediately when rate-limited |
| **Orange dot + message** | Clear UI feedback when rate-limited vs. a real error |
